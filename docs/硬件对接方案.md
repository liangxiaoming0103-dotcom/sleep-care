# 硬件对接方案

> 版本：v1.0 | 2026-07-01 | 实训项目

---

## 一、背景与目标

### 1.1 现状

当前阶段使用**后台模拟数据生成器**按生理规则自动生成虚拟睡眠数据：

```
node-cron（每日 02:00）→ seededRandom(seed) → 基础指标 + 空 JSON 字段
  → GET /api/sleep/stages  触发生成分期数据
  → GET /api/sleep/noise   触发噪音数据
  → GET /api/sleep/heart   触发心率数据（96 点）
  → GET /api/sleep/breath  触发呼吸数据（96 点）
```

虚拟设备序列号格式 `VIR` + 16 位随机字符，`is_virtual=1`，默认在线。

### 1.2 目标

真实硬件就绪后，**仅需替换数据源**和**控制指令下发模块**，其余业务逻辑（报告生成、医生授权、趋势分析）不变。

### 2.2 数据采集规范

硬件上报 JSON 格式：

```json
{
  "device_id": "DEV20260701001",
  "timestamp": "2026-07-01T03:00:00Z",
  "heart_rate": 68.5,
  "breath_rate": 14.2,
  "noise_db": 32.1,
  "sleep_stage": 2,
  "battery": 85,
  "rssi": -42
}
```

| 字段 | 类型 | 单位 | 范围 | 采集频率 |
|------|------|------|------|---------|
| `heart_rate` | float | bpm | 40-120 | 每 60 秒 |
| `breath_rate` | float | 次/分 | 8-25 | 每 60 秒 |
| `noise_db` | float | dB SPL | 20-80 | 每 10 秒 |
| `sleep_stage` | int | — | 0-3 | 每 30 秒 |
| `battery` | int | % | 0-100 | 每 5 分钟 |
| `rssi` | int | dBm | -90 ~ -30 | 每 5 分钟 |

### 2.3 写入策略

```
原始数据 → 内存环形缓冲（1024 条）
  → 两个刷入条件（满足任一即写）：
      a) 缓冲满 100 条
      b) 距上次刷入已过 5 分钟
  → 批量 INSERT INTO sensor_data_raw
  → 每日凌晨 03:00 聚合写入 sleep_reports（日报告）
```

| 表 | 用途 | 保留 |
|------|------|------|
| `sensor_data_raw` | 硬件原始高频数据 | 90 天 |
| `sleep_reports` | 每日聚合报告 | 永久 |

### 2.4 新增表结构

```sql
CREATE TABLE sensor_data_raw (
  id           BIGINT NOT NULL AUTO_INCREMENT,
  device_id    INT NOT NULL,
  timestamp    DATETIME NOT NULL,
  heart_rate   DECIMAL(5,1) NULL,
  breath_rate  DECIMAL(4,1) NULL,
  noise_db     DECIMAL(4,1) NULL,
  sleep_stage  TINYINT NULL COMMENT '0=清醒 1=浅睡 2=深睡 3=REM',
  battery      TINYINT NULL,
  rssi         SMALLINT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_device_time (device_id, timestamp),
  KEY idx_device (device_id),
  KEY idx_time (timestamp)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.5 MQTT Subscriber 伪代码

```javascript
// backend/mqtt/subscriber.js
const mqtt = require('mqtt');
const { getDb, saveDb } = require('../db/connection');

const buffer = new Map(); // device_id → rows[]

const client = mqtt.connect('mqtt://localhost:1883');

client.on('connect', () => {
  client.subscribe('sleep/+/data');
  client.subscribe('sleep/+/heartbeat');
});

client.on('message', (topic, payload) => {
  const data = JSON.parse(payload.toString());
  const deviceId = topic.split('/')[1];

  if (topic.endsWith('/data')) {
    // 写入缓冲
    if (!buffer.has(deviceId)) buffer.set(deviceId, []);
    buffer.get(deviceId).push(data);
    // 满 100 条或 5 分钟刷入
    if (buffer.get(deviceId).length >= 100) flush(deviceId);
  }

  if (topic.endsWith('/heartbeat')) {
    // 更新心跳
    db.run('UPDATE devices SET online_status=1, last_heartbeat=NOW() WHERE id=?', [deviceId]);
  }
});

setInterval(() => {
  for (const [deviceId] of buffer) flush(deviceId);
}, 5 * 60 * 1000);
```

## 三、设备绑定与配网

### 3.1 虚拟 → 真实绑定流程

**现有**：
```
点击「添加演示设备」→ 自动生成 VIR+16位 → is_virtual=1
```

**改造后，三种绑定方式**：

#### 方式 A：扫码绑定（推荐）

```
1. 设备底部贴二维码 → 内容：{ device_serial: "DEV20260701001", model: "SM-01" }
2. 小程序 wx.scanCode() 扫码 → 获取 device_serial
3. POST /api/device/bind { device_serial }
4. 后端验证：设备未被绑定 → 写入 devices 表（is_virtual=0）
5. 绑定成功，设备卡片出现在列表中
```

#### 方式 B：蓝牙配网

```
1. 小程序 wx.openBluetoothAdapter() → 搜索设备（名称 "SleepCare-XXXX"）
2. 连接设备蓝牙 → 发送 Wi-Fi SSID + 密码
3. 设备连接 Wi-Fi → 注册到 MQTT Broker → 首次上报 device_id
4. 云端自动识别用户 openId → 绑定到当前账号
```

#### 方式 C：手动输入

```
输入设备底部 16 位序列号 → POST /api/device/add { device_serial, is_virtual: false }
  → 复用现有接口，仅改参数
```

### 3.2 设备心跳与在线状态

```
硬件每 60 秒 → MQTT sleep/{device_id}/heartbeat
  → { device_id, battery, rssi, firmware_version, timestamp }

后端逻辑：
  - 收到心跳 → UPDATE devices SET online_status=1, last_heartbeat=NOW(), battery=x
  - 定时任务（每 2 分钟）→ UPDATE devices SET online_status=0
    WHERE last_heartbeat < NOW() - INTERVAL 2 MINUTE
```

### 3.3 devices 表扩展字段

```sql
ALTER TABLE devices ADD COLUMN last_heartbeat    DATETIME NULL;
ALTER TABLE devices ADD COLUMN firmware_version  VARCHAR(16) NULL;
ALTER TABLE devices ADD COLUMN battery_level     TINYINT NULL;
ALTER TABLE devices ADD COLUMN model             VARCHAR(32) NULL DEFAULT 'SM-01';
```

### 3.4 设备离线通知

| 事件 | 检测方式 | 用户通知 |
|------|---------|---------|
| 设备离线 | 心跳超时 2 分钟 | 小程序首页卡片变灰 + 🔴 红点 |
| 设备恢复 | 心跳恢复 | 卡片恢复绿色，红点消失 |
| 低电量 | battery < 15% | 微信服务通知「请为睡眠监测垫充电」 |

## 四、部署架构

### 4.1 服务拓扑

```
                          ┌──────────────┐
                          │   用户请求    │
                          └──────┬───────┘
                                 │
                          ┌──────▼──────┐
                          │   Nginx     │  反向代理 + SSL + 限流
                          │  :443/:80   │
                          └──────┬──────┘
                 ┌───────────────┼───────────────┐
                 │               │               │
          ┌──────▼──────┐ ┌─────▼─────┐  ┌──────▼──────┐
          │  App #1     │ │  App #2   │  │  App #3     │  PM2 Cluster
          │  Express    │ │  Express  │  │  Express    │  (3 实例)
          │  :3000      │ │  :3001   │  │  :3002      │
          └──────┬──────┘ └─────┬─────┘  └──────┬──────┘

                         │ MySQL 8.0   │  主库（读写）
```

### 4.3 性能预估

| 指标 | 100 设备 | 1000 设备 | 10000 设备 |
|------|---------|----------|-----------|
| 心跳 QPS | 1.7 | 17 | 170 |
| 数据写入 QPS | 3.3 | 33 | 330 |
| API 请求 QPS | ~5 | ~50 | ~500 |
| 数据库连接池 | 10 | 30 | 80 |
| 日增数据量 | ~2MB | ~20MB | ~200MB |

### 4.4 高可用策略

| 层面 | 策略 |
|------|------|
| 应用层 | PM2 Cluster 多实例 + Nginx upstream 负载均衡 |
| 数据库 | MySQL 主从复制（1 主 2 从），读写分离 |
| 消息队列 | MQTT Broker 集群（EMQX 支持） |
| 故障转移 | 主库宕机 → 从库自动提升（MHA / Orchestrator） |
| 数据备份 | 每日凌晨全量备份 + binlog 增量备份，保留 30 天 |

### 4.5 数据归档

| 数据 | 热存储 | 归档 | 清理 |
|------|--------|------|------|
| `sensor_data_raw` | 90 天 | 导出 CSV → OSS | DROP 过期分区 |
| `sleep_reports` | 永久 | 归档表 `_archive` | 手动 |
| 日志 | 7 天 | — | logrotate 按天 |

### 4.6 监控告警

| 指标 | 阈值 | 通知方式 |
|------|------|---------|
| 设备离线率 > 20% | 持续 5 分钟 | 钉钉群机器人 |
| API 响应 P99 > 3s | 持续 10 分钟 | 企业微信 |
| MySQL 连接数 > 80% | 即时 | 邮件 |
| MQTT Broker 断开 | 即时 | 短信 |
| 磁盘 > 85% | 即时 | 邮件 |
| 服务宕机 | 即时 | PM2 + 钉钉 |

**工具链**：PM2（进程守护）+ Prometheus + Grafana（监控面板）+ ELK（日志中心）。

## 六、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| MQTT Broker 宕机 | 中 | 高 | 集群部署 + 硬件本地缓存 |
| 数据库写入瓶颈 | 低 | 高 | 缓冲批量写入 + 分区表 |
| 硬件固件兼容性 | 中 | 中 | 定义最低固件版本号 + OTA 升级通道 |
| Wi-Fi 配网失败率高 | 高 | 低 | 蓝牙辅助配网 + 错误码提示 |
| 数据迁移丢数据 | 低 | 高 | 逐表校验 + 抽样对比 + 回滚方案 |

---

> **核心原则**：Express + MySQL（或 SQLite）是强制技术栈。硬件接入后仅需新增 MQTT Subscriber 模块和 sensor_data_raw 表，现有业务 API 无需改动。
