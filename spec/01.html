📋 三、需求概览（教师口述 + 白板简图）

3.1 项目背景

我们要为“智能睡眠环境调控设备”开发配套软件。设备可以监测睡眠、调节灯光声音，但我们先做软件部分。软件分成三块：

  • 微信小程序：普通用户使用，看睡眠报告、管理设备、授权医生
  • 云后台API：处理数据、存储、模拟数据生成
  • 医生端Web：医生登录后查看患者的睡眠数据

🔑 核心设计原则 数据库优先——我们先设计好数据模型（表结构），再围绕它开发API和界面。这样后续接入真实硬件时，只需换数据源，API和前端几乎不用改。

3.2 MVP功能范围（板书）

  • ✅ 用户注册/登录（手机号+密码，bcrypt加密）
  • ✅ 虚拟设备管理（添加、解绑、修改昵称）
  • ✅ 睡眠报告：首页卡片、睡眠分期柱状图、环境噪音曲线图、日/周/月趋势图
  • ✅ 个性化作息设置（就寝时间、起床时间）
  • ✅ 医生授权：用户侧授权/撤销，医生侧登录查看患者报告（只读）
  • ⏸️ 暂不实现（可预留）：e-CBTi全部模块、真实硬件控制、OTA升级、消息推送

3.3 五张核心数据表（板书/投影）

根据需求文档，我们抽象出 5张核心表，所有业务数据都存储在这些表中：

关键 这些表结构完全对齐需求文档，后续接入真实硬件时数据字段无需变更。

3.4 无硬件实现策略（重点强调）

模块模拟方式
设备绑定输入任意16位序列号或点“添加演示设备”，后台生成虚拟ID
睡眠数据后台定时任务根据用户作息自动生成假数据，符合生理规律，存入 sleep_reports 表
环境调控小程序里滑块调亮度音量，弹窗提示“硬件未就绪”
OTA升级点击检查更新，返回模拟版本号，不真正升级

⚙️ 四、技术架构与AI辅助开发

4.1 技术栈（白板书写）

  • 后端：Node.js + Express（轻量快速）
  • 数据库：SQLite（文件数据库，零配置，便于实训）→ 后续可无缝迁移至 MySQL/PostgreSQL
  • 小程序：原生框架 + WeUI组件库
  • 医生端Web：原生HTML/CSS/JS + ECharts
  • 版本控制：Git（通过VSCode SCM面板操作）
  • AI辅助：Claude Code（终端启动，可自动创建文件）

4.2 简单架构图（教师手绘白板）

[小程序] ──API──> [云后台API (Express)] ──> [数据库(SQLite)]
[医生Web] ──API──> [云后台API] ──────────────┘

数据流向：用户在小程序操作 → 调用后台API → 后台读写数据库 → 返回结果。医生Web同理。

✅ 任务3：用 Claude 自动生成 .gitignore 文件 ⏱️ 8分钟
📚 前置知识：.gitignore 文件是做什么的？
.gitignore 文件告诉 Git 哪些文件/文件夹不需要被版本控制。
为什么要忽略某些文件？
  • node_modules/：体积巨大，且可通过 package.json 重新安装
  • 日志文件（*.log）：每台电脑的日志都不同，存到仓库没有意义
  • 操作系统临时文件：如 macOS 的 .DS_Store、Windows 的 Thumbs.db
  • 环境变量文件（.env）：含敏感信息，绝对不能提交
  1. 在 VSCode 中打开终端
    按下 Ctrl+`（反引号）打开集成终端。
  2. 启动 Claude Code
    在终端中输入 claude 并按回车。等待几秒，直到看到提示符（如 > ）。 ⚠️ 首次运行 可能会要求登录或授权，按终端提示操作。
  3. 让 AI 自动创建 .gitignore 文件
    在 Claude 交互界面中输入以下提示词并回车：
    📝 提示词：
    请为 Node.js + 微信小程序混合项目生成标准的.gitignore文件，忽略node_modules、日志、操作系统临时文件、VSCode配置、微信开发者工具临时目录、环境变量文件等，并直接写入项目根目录的 .gitignore 文件中。
    AI 会自动创建 .gitignore 文件并写入内容。
  4. 通过 SCM 面板暂存并提交
    • 点击左侧“源代码管理”图标，.gitignore 会出现在 “未跟踪” 分组下
    • 点击文件右侧出现的 “+”号（暂存更改）
    • 在 SCM 面板顶部输入框中输入提交消息：chore: 添加gitignore
    • 点击 “✔”提交（Commit） 按钮
🔍 内容走读：理解 AI 生成的 .gitignore
# 依赖目录 - 体积最大，必须忽略 node_modules/ # 日志文件 - 每台机器不同 *.log # 操作系统生成的文件 .DS_Store # macOS 文件夹元数据 Thumbs.db # Windows 缩略图缓存 # VSCode 配置（个人偏好） .vscode/ # 环境变量文件（含敏感信息） .env .env.local # 微信开发者工具临时目录 .wx_temp/ miniprogram_npm/
  • node_modules/：最重要的忽略项，运行 npm install 时自动生成
  • .env：存放数据库密码、API密钥等敏感信息，绝对不能提交到 Git

💡 重点理解 .gitignore 的核心原则：只提交“源代码”和“配置文件”，不提交“生成的文件”和“敏感信息”。

🔍 内容走读：理解 AI 生成的数据库设计（ER图 + DDL）

1. ER图（Mermaid）

AI 生成的 ER 图代码大致如下（以 users 和 devices 为例）：

erDiagram users ||--o{ devices : "拥有" users ||--o{ sleep_reports : "生成" users ||--o{ doctor_authorizations : "发起授权" users ||--o{ doctor_authorizations : "处理授权" users { integer id PK "主键，自增" varchar phone UK "手机号，11位，唯一" varchar password_hash "密码哈希，bcrypt" varchar nickname "昵称" varchar role "角色：patient/doctor/admin" integer status "状态：1启用 0禁用" datetime created_at "创建时间" datetime updated_at "更新时间" } devices { integer id PK "主键，自增" varchar serial_no UK "设备序列号，VIR+16位随机" integer user_id FK "所属用户ID → users.id" varchar nickname "设备昵称" integer is_virtual "是否虚拟设备：1是 0否" integer online_status "在线状态：1在线 0离线" datetime created_at "绑定时间" datetime updated_at "更新时间" }
  • ||--o{：表示“一对多”关系。一个用户可拥有多台设备
  • PK：主键，唯一标识一条记录
  • FK：外键，指向另一张表的主键（如 user_id → users.id）
  • UK：唯一键（如手机号、设备序列号）
  • 五张表的关系：users 是核心，通过外键与 devices、sleep_reports、user_settings（一对一）、doctor_authorizations（作为患者和医生）关联

2. DDL 建表语句（SQLite）

-- users 表 CREATE TABLE users ( id INTEGER PRIMARY KEY AUTOINCREMENT, phone VARCHAR(11) NOT NULL UNIQUE, password_hash VARCHAR(255) NOT NULL, nickname VARCHAR(50), role VARCHAR(20) DEFAULT 'patient', status INTEGER DEFAULT 1, created_at DATETIME DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ); -- devices 表（外键关联 users） CREATE TABLE devices ( id INTEGER PRIMARY KEY AUTOINCREMENT, serial_no VARCHAR(20) NOT NULL UNIQUE, user_id INTEGER NOT NULL, nickname VARCHAR(50) DEFAULT '我的设备', is_virtual INTEGER DEFAULT 1, online_status INTEGER DEFAULT 1, created_at DATETIME DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE ); -- sleep_reports 表 CREATE TABLE sleep_reports ( id INTEGER PRIMARY KEY AUTOINCREMENT, user_id INTEGER NOT NULL, report_date DATE NOT NULL, sleep_score INTEGER, total_minutes INTEGER, deep_minutes INTEGER, light_minutes INTEGER, rem_minutes INTEGER, awake_minutes INTEGER, wake_count INTEGER, heart_rate_json TEXT, breath_rate_json TEXT, noise_json TEXT, sleep_stages_json TEXT, abnormal_events_json TEXT, created_at DATETIME DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE, UNIQUE(user_id, report_date) ); -- user_settings 表（一对一） CREATE TABLE user_settings ( id INTEGER PRIMARY KEY AUTOINCREMENT, user_id INTEGER NOT NULL UNIQUE, bed_time TIME, wake_time TIME, sunrise_duration_minutes INTEGER DEFAULT 10, sound_type VARCHAR(20), brightness_level INTEGER DEFAULT 5, volume_level INTEGER DEFAULT 5, timezone VARCHAR(50) DEFAULT 'Asia/Shanghai', dnd_enabled INTEGER DEFAULT 0, dnd_start TIME, dnd_end TIME, created_at DATETIME DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE ); -- doctor_authorizations 表 CREATE TABLE doctor_authorizations ( id INTEGER PRIMARY KEY AUTOINCREMENT, patient_id INTEGER NOT NULL, doctor_id INTEGER NOT NULL, status VARCHAR(20) DEFAULT 'pending', expire_date DATE, doctor_note TEXT, requested_at DATETIME DEFAULT CURRENT_TIMESTAMP, responded_at DATETIME, created_at DATETIME DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (patient_id) REFERENCES users(id) ON DELETE CASCADE, FOREIGN KEY (doctor_id) REFERENCES users(id) ON DELETE CASCADE );
  • PRIMARY KEY AUTOINCREMENT：主键自增，每插入一条新记录自动生成唯一ID
  • NOT NULL：字段不能为空（如手机号、密码哈希必须填写）
  • UNIQUE：字段值不能重复（如手机号、序列号）
  • DEFAULT：默认值（如 role DEFAULT 'patient'，新用户默认为患者角色）
  • FOREIGN KEY ... REFERENCES：外键约束，确保数据完整性。例如 devices.user_id 必须指向一个有效的 users.id
  • ON DELETE CASCADE：当用户被删除时，自动删除其所有关联数据（设备、报告、设置等），保证数据一致性
  • UNIQUE(user_id, report_date)：复合唯一约束，同一用户同一天只能有一条报告
  • TEXT 类型用于存储 JSON 数据（如 sleep_stages_json、noise_json）

💡 重点理解 DDL 是数据库的“施工图”。有了这些建表语句，第2节我们就能直接创建数据库，把设计变成可用的存储结构。这套表结构完全对齐需求文档，后续接入真实硬件时，只需更换数据库连接，表和字段无需改动。

❗ 七、常见问题及解决方案（教师备用）

❌ 问题1：终端输入 claude 提示“命令未找到”

原因：Claude Code 扩展未安装或未正确配置 PATH。

解决方案：确认已在 VSCode 中安装 Claude Code 扩展；安装后重启 VSCode；如仍不行，尝试 npx @anthropic-ai/claude-code。

❌ 问题2：AI 没有自动创建文件，只是在终端输出了内容

原因：提示词中未明确要求“写入文件”。

解决方案：检查提示词是否包含“写入到 xxx 文件”或“保存为 xxx”的明确指令。如果仍然只输出文本，可手动复制并新建文件粘贴保存。

❌ 问题3：学生不理解 DDL 中的外键约束

原因：第一次接触数据库设计，不熟悉。

解决方案：

  • 在白板上画出五张表的关联图，用箭头标出外键指向
  • 用“用户-设备”的例子解释：一个用户有多台设备，所以 devices.user_id 指向 users.id
  • 强调 ON DELETE CASCADE 的含义：删除用户时自动删除其所有数据，保证数据一致性

❌ 问题4：学生不理解“为什么今天就要设计数据库”

原因：习惯了“先写代码再想数据”的开发方式。

解决方案：

  • 强调“数据库优先”的核心理念：先确定数据模型，再围绕它开发 API
  • 类比：盖房子先画图纸（ER图）再施工（写代码）
  • 强调第2节就要用今天的 DDL 创建数据库，如果今天没设计好，下节课就无法继续

❌ 问题5：/init 命令生成的 CLAUDE.md 内容不完整

原因：/init 生成的是基础模板，需要人工补充规范内容。

解决方案：提醒学生使用教师提供的规范模板进行合并，可以通过 Claude 提示词“请将以下规范内容合并到 CLAUDE.md 中”来完成，或者手动复制粘贴。
