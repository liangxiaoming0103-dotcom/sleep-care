3.1 JWT 认证中间件复习（约3分钟）

💡 教师口述
上节课我们实现了登录接口，它会返回一个 JWT 令牌给客户端。但问题是：客户端后续请求怎么证明自己已经登录了？

答案就是：客户端在每次请求的 HTTP Header 中带上这个 Token，格式为：
Authorization: Bearer <token>

服务端需要一个“认证中间件”——它会在请求到达具体接口之前，先拦截下来，做三件事：
1. 从请求头中提取 Token
2. 验证 Token 的签名是否有效（用我们保存的 JWT_SECRET）
3. 如果有效，把 Token 中存储的用户信息（如 id、phone）挂载到 req.user 上，放行请求；如果无效，直接返回 401 或 403。

在第2节中，我们已经用 Claude 生成了 authenticateToken 中间件。今天我们会直接在设备接口中使用它。
💡 关键 req.user 中包含 id（用户ID），后续设备操作中我们需要用这个 id 来关联设备，实现数据隔离。

🔍 代码走读：已有的 authenticateToken 中间件
// JWT认证中间件（已在 app.js 中） function authenticateToken(req, res, next) { // 1. 从请求头获取 Authorization const authHeader = req.headers['authorization']; const token = authHeader && authHeader.split(' ')[1]; // 2. 如果没有 Token if (!token) { return res.status(401).json({ code: 401, message: '未提供认证令牌', data: null }); } // 3. 验证 Token jwt.verify(token, JWT_SECRET, (err, decoded) => { if (err) { return res.status(403).json({ code: 403, message: '令牌无效或已过期', data: null }); } // 4. 验证通过，挂载用户信息到 req.user req.user = decoded; next(); }); }
  • req.headers['authorization']：从请求头中获取 Authorization 字段。
  • authHeader && authHeader.split(' ')[1]：按空格分割，取第二部分（即 Token 本身）。
  • jwt.verify(token, JWT_SECRET, ...)：验证 Token 签名和有效期。验证通过后，decoded 就是 JWT payload（包含 id、phone、role）。
  • req.user = decoded：将用户信息挂载到请求对象上。后续路由通过 req.user.id 获取当前用户ID。

💡 重点理解 这个中间件是“受保护路由”的守门员。所有需要登录才能操作的接口（如设备管理），都要在路由定义中作为第二个参数传入 authenticateToken。

🔍 代码走读：设备管理接口（SQLite版，基于 sql.js）

1. 列表接口 (GET /list)

app.get('/api/device/list', authenticateToken, async (req, res) => { const userId = req.user.id; const db = await getDb(); // sql.js 的 getDb() 是异步的 const devices = dbGetAll( db, 'SELECT * FROM devices WHERE user_id = ? ORDER BY created_at DESC', [userId] ); res.json({ code: 0, message: 'success', data: devices }); });
  • authenticateToken：保护路由，确保只有登录用户能访问。
  • await getDb()：sql.js 的 getDb() 返回 Promise，需要使用 await 等待 WASM 加载完成。
  • dbGetAll：辅助函数，执行 SELECT 并返回所有行（已在 app.js 中定义）。
  • ORDER BY created_at DESC：最新添加的设备排在前面。
  • 注意：查询操作不需要调用 saveDb()，因为数据没有修改。

2. 添加设备 (POST /add) — 显式持久化

app.post('/api/device/add', authenticateToken, async (req, res) => { const { device_serial, is_virtual } = req.body; const userId = req.user.id; const db = await getDb(); let serialNo; // ... 序列号生成逻辑 ... const now = new Date().toISOString(); try { db.run( `INSERT INTO devices (user_id, serial_no, nickname, is_virtual, online_status, created_at, updated_at) VALUES (?, ?, ?, ?, ?, ?, ?)`, [userId, serialNo, '我的设备', is_virtual ? 1 : 0, 1, now, now] ); // ⚠️ 关键步骤：sql.js 必须显式调用 saveDb() 持久化到磁盘 saveDb(); const newDevice = dbGetOne(db, 'SELECT * FROM devices WHERE serial_no = ?', [serialNo]); res.json({ code: 0, message: '添加成功', data: newDevice }); } catch (err) { if (err.message && err.message.includes('UNIQUE constraint failed')) { return res.json({ code: 1001, message: '该序列号已被绑定' }); } console.error('[DB] add device error:', err); res.json({ code: 1001, message: '添加失败' }); } });
  • db.run()：执行 INSERT 操作，参数化查询防止 SQL 注入。
  • saveDb()：关键步骤——sql.js 的数据在内存中，调用 saveDb() 将内存数据库导出为二进制数据并写入 sleep_care.db 文件。如果忘记调用，数据会在服务器重启后丢失。
  • UNIQUE constraint failed：捕获序列号冲突异常，返回友好错误。

3. 修改设备 (PUT /devices/:id)

app.put('/api/devices/:id', authenticateToken, async (req, res) => { const deviceId = parseInt(req.params.id, 10); const { nickname } = req.body; const userId = req.user.id; const db = await getDb(); // 权限校验：验证设备属于当前用户 const device = dbGetOne(db, 'SELECT * FROM devices WHERE id = ?', [deviceId]); if (!device) return res.json({ code: 1001, message: '设备不存在' }); if (device.user_id !== userId) return res.json({ code: 1001, message: '无权操作该设备' }); const now = new Date().toISOString(); db.run('UPDATE devices SET nickname = ?, updated_at = ? WHERE id = ?', [nickname.trim(), now, deviceId]); saveDb(); // 持久化更新 const updated = dbGetOne(db, 'SELECT * FROM devices WHERE id = ?', [deviceId]); res.json({ code: 0, message: '修改成功', data: updated }); });
  • 路径参数 :id：使用设备主键作为标识，符合 RESTful 规范。
  • 权限校验：先查询设备，检查 user_id 是否匹配当前用户，防止越权操作。
  • saveDb()：UPDATE 后同样需要调用，确保磁盘文件同步更新。

4. 删除设备 (DELETE /devices/:id)

app.delete('/api/devices/:id', authenticateToken, async (req, res) => { const deviceId = parseInt(req.params.id, 10); const userId = req.user.id; const db = await getDb(); const device = dbGetOne(db, 'SELECT * FROM devices WHERE id = ?', [deviceId]); if (!device) return res.json({ code: 1001, message: '设备不存在' }); if (device.user_id !== userId) return res.json({ code: 1001, message: '无权操作该设备' }); db.run('DELETE FROM devices WHERE id = ?', [deviceId]); saveDb(); // 持久化删除 res.json({ code: 0, message: '删除成功', data: null }); });
  • 硬删除：直接从数据库中移除记录。
  • 权限校验：与修改接口相同，先验证所有权。
  • saveDb()：DELETE 后调用，确保磁盘数据与内存一致。

💡 重点理解 所有数据库操作都是 用户绑定的——通过 user_id 过滤，确保用户只能操作自己的设备。这是多租户系统的基本设计原则。

⚠️ 关于 saveDb()：在 sql.js 中，saveDb() 不是自动调用的。每次 INSERT、UPDATE、DELETE 后都必须显式调用，否则数据只存在于内存中，重启即丢失。这是与 better-sqlite3 最大的不同之处。

  2. Body → raw → JSON：{"phone":"13800138000","password":"123456"}
  3. 点击 Send，从响应中复制 token 值（一长串字符串）。

6.2 第二步：设置 Authorization 请求头

  1. 新建请求（如 GET 设备列表），切换到 Headers 选项卡。
  2. 添加一行：Key = Authorization，Value = Bearer 你复制的token
  3. ⚠️ 注意 Bearer 和 token 之间有一个空格，不要加引号。
📚 解析：Authorization: Bearer <token>
  • Authorization：HTTP 标准请求头字段，专门用来传递认证凭证。
  • Bearer：认证方案，表示“持有者令牌”——谁持有这个令牌，谁就拥有对应权限。
  • <token>：JWT 令牌本身，包含用户身份信息。

❗ 八、常见问题及解决方案

❌ 问题1：小程序请求接口报错 "request:fail url not in domain list"

原因：小程序默认要求请求地址为 HTTPS，且需配置合法域名。

解决方案：在微信开发者工具中，点击“详情” → “本地设置” → 勾选“不校验合法域名”。

❌ 问题2：Postman 请求设备接口返回 401 "未提供认证令牌"

原因：没有在请求头中添加 Authorization。

解决方案：先调用登录接口获取 token，然后在 Postman 的 Headers 中添加 Authorization: Bearer <token>。

❌ 问题3：设备列表接口返回空数组

原因：数据库中没有设备记录，或当前用户没添加过设备。

解决方案：先调用添加设备接口（POST /add），然后重新查看列表。

❌ 问题4：添加设备时提示 "该序列号已被绑定"

原因：手动输入的序列号在数据库中已存在，或虚拟设备生成时碰撞（概率极低）。

解决方案：如果是手动输入，换一个序列号；如果是虚拟设备，重试添加（代码中已做3次重试）。

❌ 问题5：小程序登录后没有跳转，也没有报错

原因：可能是 wx.navigateTo 的路径写错了，或 pages/devices/devices 页面不存在。

解决方案：检查 app.json 中的 pages 配置，确保路径正确。

❌ 问题6：重启服务器后设备数据丢失

原因：没有调用 saveDb()，或 saveDb() 实现有误（sql.js 需要显式持久化）。

解决方案：

  1. 检查 app.js 中每个 INSERT/UPDATE/DELETE 操作后是否调用了 saveDb()
  2. 检查 db/connection.js 中 saveDb() 是否正确使用 db.export() 和 fs.writeFileSync()
  3. 如果代码已正确但数据仍丢失，在终端中查看 saveDb() 是否有报错（如权限问题）

❌ 问题7：微信开发者工具没有自动创建 login/devices 文件夹

原因：修改 app.json 后没有保存，或 app.json 格式有误。

解决方案：确保 app.json 文件已保存，检查 JSON 格式是否合法，点击“编译”按钮强制重新编译。

❌ 问题8：代码中使用了 dbGetAll 但未定义

原因：app.js 中未定义 dbGetAll 辅助函数。

解决方案：在 app.js 中补充定义（与 dbGetOne 类似，但循环获取所有行）：

function dbGetAll(db, sql, params = []) { const stmt = db.prepare(sql); stmt.bind(params); const rows = []; while (stmt.step()) { rows.push(stmt.getAsObject()); } stmt.free(); return rows; }
