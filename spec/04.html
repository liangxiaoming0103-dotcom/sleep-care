📖 一、课程导入

🎤 教师口述（3分钟）
同学们，上节课我们完成了设备管理API的SQLite持久化，并实现了小程序登录页和设备管理页。今天我们要做一个更有趣的事情：生成睡眠数据并将其持久化到数据库，然后在首页展示。

睡眠数据是你们产品的“核心资产”——没有数据，睡眠报告、趋势分析、医生诊断都无从谈起。但由于我们还没有真实硬件，我们需要模拟数据来验证整个流程。但模拟数据不能每次请求都重新生成，否则用户看到的数据会“飘忽不定”，而且无法用于历史趋势分析。

今天的核心设计：实现一个“查询-生成-插入”模式——
  1. 用户请求某一天的睡眠报告
  2. 后端先查询 sleep_reports 表，看是否已有数据
  3. 如果有，直接返回
  4. 如果没有，生成模拟数据，存入数据库（调用 saveDb()），然后返回
这样，数据一旦生成就永远存在，重启服务器也不会丢失。这才是真正的“持久化”。

3.2 “查询-生成-插入”模式（约4分钟）

💡 教师口述
我们的 API 设计遵循一个重要的原则：数据生成一次，永久存储。

传统做法（错误）：每次请求都重新生成模拟数据，导致数据不可复现、无法做历史分析。

我们的做法（正确）：
  1. 客户端请求 GET /api/sleep/report/daily?date=2026-06-19
  2. 服务端执行 SELECT * FROM sleep_reports WHERE user_id=? AND device_id=? AND report_date=?
  3. 如果查到记录 → 直接返回数据库中的数据
  4. 如果没查到 → 使用确定性伪随机算法生成模拟数据 → INSERT INTO sleep_reports → 调用 saveDb() 持久化 → 然后返回新插入的记录
这样，每个用户每天的睡眠数据在第一次请求时生成，之后永远固定。这为后续的日/周/月趋势分析、医生查看历史数据提供了坚实的基础。

⚠️ 关键点（sql.js）：由于我们使用 sql.js，数据库操作在内存中进行，必须显式调用 saveDb() 才能将更改写入 sleep_care.db 磁盘文件。这是与 better-sqlite3 最大的不同，也是学生最容易遗漏的步骤。

🔍 代码走读：睡眠报告API（“查询-生成-插入”模式，基于 sql.js）

1. 确定性随机函数（与第3节相同）

function seededRandom(seed) { let s = 0; for (let i = 0; i < seed.length; i++) { s = (s * 31 + seed.charCodeAt(i)) & 0x7fffffff; } return function() { s = (s * 1103515245 + 12345) & 0x7fffffff; return (s >>> 0) / 2147483647; }; }

2. 核心逻辑：先查后插（含 saveDb()）

app.get('/api/sleep/report/daily', authenticateToken, async (req, res) => { const userId = req.user.id; let dateStr = req.query.date; if (!dateStr) { const yesterday = new Date(); yesterday.setDate(yesterday.getDate() - 1); dateStr = yesterday.toISOString().split('T')[0]; } const db = await getDb(); // 1. 获取用户第一台设备（简化处理） let deviceId = 0; const deviceRow = dbGetOne(db, 'SELECT id FROM devices WHERE user_id = ? LIMIT 1', [userId]); if (deviceRow) deviceId = deviceRow.id; // 2. 先查询数据库 let report = dbGetOne( db, 'SELECT * FROM sleep_reports WHERE user_id = ? AND device_id = ? AND report_date = ?', [userId, deviceId, dateStr] ); if (report) { // 3. 存在则直接返回 return res.json({ code: 0, message: 'success', data: report }); } // 4. 不存在则生成模拟数据 const seedKey = `${userId}_${deviceId}_${dateStr}`; const rand = seededRandom(seedKey); const total_sleep_minutes = Math.floor(300 + rand() * 180); const deep_ratio = 0.15 + rand() * 0.20; const rem_ratio = 0.20 + rand() * 0.05; const deep_sleep_minutes = Math.floor(total_sleep_minutes * deep_ratio); const rem_sleep_minutes = Math.floor(total_sleep_minutes * rem_ratio); const light_sleep_minutes = total_sleep_minutes - deep_sleep_minutes - rem_sleep_minutes; const sleep_score = Math.floor(60 + rand() * 40); const awake_count = Math.floor(rand() * 6); const awake_minutes = Math.floor(rand() * 30); // 0-30分钟清醒 // 5. 插入数据库（JSON字段暂存空数组） try { db.run( `INSERT INTO sleep_reports (user_id, device_id, report_date, sleep_score, total_sleep_minutes, deep_sleep_minutes, light_sleep_minutes, rem_sleep_minutes, awake_minutes, awake_count, heart_rate_json, sleep_stages_json, noise_json) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)`, [userId, deviceId, dateStr, sleep_score, total_sleep_minutes, deep_sleep_minutes, light_sleep_minutes, rem_sleep_minutes, awake_minutes, awake_count, '[]', '[]', '[]'] ); // ⚠️ 关键：sql.js 必须显式调用 saveDb() 持久化到磁盘 saveDb(); // 6. 查询新插入的记录 report = dbGetOne( db, 'SELECT * FROM sleep_reports WHERE user_id = ? AND device_id = ? AND report_date = ?', [userId, deviceId, dateStr] ); return res.json({ code: 0, message: 'success', data: report }); } catch (err) { if (err.message && err.message.includes('UNIQUE constraint failed')) { // 并发场景下可能重复插入，重新查询即可 report = dbGetOne( db, 'SELECT * FROM sleep_reports WHERE user_id = ? AND device_id = ? AND report_date = ?', [userId, deviceId, dateStr] ); if (report) { return res.json({ code: 0, message: 'success', data: report }); } } console.error('[DB] sleep report error:', err); return res.json({ code: 1001, message: '生成报告失败' }); } });
  • 先查后插：避免重复生成，确保数据一致性。
  • UNIQUE(user_id, device_id, report_date)：数据库层面保证同一用户同一设备同一天只有一条记录。
  • saveDb()：关键步骤——sql.js 的数据在内存中，调用 saveDb() 将内存数据库导出为二进制数据并写入 sleep_care.db 文件。如果忘记调用，数据会在服务器重启后丢失。
  • JSON 字段：暂存 '[]'，为第5、6节分期和噪音数据预留位置。
  • 并发处理：捕获 UNIQUE 约束异常，如果插入失败但记录已存在，重新查询返回。
  • 设备关联：通过查询用户的第一台设备实现数据关联，支持多设备场景（后续可扩展为指定设备）。

💡 重点理解 这个模式是“数据持久化”的核心：数据一旦生成，永久保存。用户刷新页面不会看到数据变化，重启服务器数据依然存在。这为后续的趋势分析、医生查看历史数据提供了可靠的基础。

⚠️ 与 better-sqlite3 的区别：在 better-sqlite3 中，db.run() 会直接写入磁盘，无需额外调用。而在 sql.js 中，所有操作先在内存进行，必须显式调用 saveDb() 才能持久化。这是教学中要反复强调的易错点。

📚 前置知识：页面生命周期与下拉刷新
小程序页面有两个关键生命周期方法：
  • onLoad()：页面加载时触发，只执行一次。适合做初始数据请求。
  • onShow()：页面每次显示时触发（包括从后台切回前台）。适合刷新数据。
  • onPullDownRefresh()：用户下拉刷新时触发。需要在 .json 文件中启用 "enablePullDownRefresh": true。

❗ 七、常见问题及解决方案

❌ 问题1：sleep_reports 表不存在

原因：db/schema.js 中未包含该表的 CREATE TABLE 语句。

解决方案：参考第1节 ER 图，让 Claude 补充建表语句到 schema.js 中，重启服务器。

❌ 问题2：第一次请求生成数据，第二次请求数据不一致

原因：种子随机函数未正确使用传入的 seed，或者查询逻辑有误（如未正确查询数据库）。

解决方案：检查 seedKey 的生成逻辑，确保包含 userId、deviceId 和 dateStr。在 Claude 中调试该函数。

❌ 问题3：首页一直显示“加载中...”

原因：请求失败或 setData 未正确执行。

解决方案：在 wx.request 的 fail 和 success 中添加 console.log，查看控制台输出。检查网络请求是否成功。

❌ 问题4：重启服务器后数据丢失

原因：API 没有调用 saveDb()，或 saveDb() 实现有误。

解决方案：

  1. 检查 app.js 中 INSERT 后是否调用了 saveDb()
  2. 检查 db/connection.js 中 saveDb() 是否正确使用 db.export() 和 fs.writeFileSync()
  3. 如果代码已正确但数据仍丢失，在终端中查看 saveDb() 是否有报错（如权限问题）

❌ 问题5：深睡比例显示为小数而不是百分比

原因：WXML 中未做乘法转换。

解决方案：在 WXML 中使用 {{ (sleepData.deep_sleep_minutes / sleepData.total_sleep_minutes * 100).toFixed(0) }}%。

❌ 问题6：查询设备时返回空，导致 device_id = 0

原因：用户尚未添加设备。这是正常情况，数据仍会存入，但建议提示用户先添加设备。

解决方案：可在前端提示“请先添加设备以获取更精准的睡眠报告”，或让 API 在无设备时自动创建一个虚拟设备（需扩展逻辑）。本节课简化处理，允许 device_id = 0。
